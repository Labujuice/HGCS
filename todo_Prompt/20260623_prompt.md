### Feedback
* **問題根源**：
  - 在 [gateway.py](../gateway/gateway.py) 的 `_handle_upload_mission` 中，我們在將前端航點列表編譯為 MAVLink 任務項目時，**硬編碼強行在 `seq 0` 插入了一個 `WAYPOINT` 作為 Home 點**，並使實際任務航點（包含 `TAKEOFF`）從 `seq 1` 開始。
  - 這在 ArduPilot 中是必要的，因為 ArduPilot 強制將 `seq 0` 當作 Home 點。
  - 然而在 **PX4** 中，**`seq 0` 並非 Home 點，而是任務的第一個執行航點**。當 PX4 可行性檢查器（Feasibility Checker）發現 `seq 0` 是普通 `WAYPOINT`，而後面卻有 `TAKEOFF` 時，便會判定「起飛點不是第一個航點項目（takeoff not first waypoint item）」進而拒絕整個任務。
* **修復方案**：
  - 在 [gateway.py](../gateway/gateway.py) 的遙測 HEARTBEAT 接收端記錄車輛的 `autopilot` 類型。
  - 在上傳任務時，動態判斷是否為 PX4 飛控。如果是 PX4，則**不強行在前置插入 `seq 0` Home 點**，而是直接將 `TAKEOFF` 以 `seq 0` 上傳給飛控，這能 100% 順利通過 PX4 可行性檢驗；對於非 PX4（如 ArduPilot）則保留 `seq 0` 作為 Home 點。

### Report

#### 1. 識別飛控類型並動態優化任務上傳序列 (Sequence 0)
- **實作結果**：
  - 在 [gateway.py](../gateway/gateway.py) 中，於 `Gateway.__init__` 初始化 `self.vehicle_autopilots` 字典，並在 `_mavlink_reader_loop` 讀取 `HEARTBEAT` 訊息時實時記錄連線載具的 `msg.autopilot` 類型。
  - 修改 `_handle_upload_mission` 航點序列對照邏輯：在將任務轉譯為 MAVLink 任務項目上傳時，動態讀取載具飛控類型是否為 PX4 (`MAV_AUTOPILOT_PX4`，值為 12)。
  - 如果為 **PX4**（包括 SITL、實機以及預載的模擬無人機），則**不再強行於 `seq 0` 前置插入 `WAYPOINT`（Home點）**，而是直接將包含 `TAKEOFF` 的任務首個項目編排為 `seq 0` 發送至飛控。對於非 PX4（如 ArduPilot）則繼續保留 `seq 0` 作為 Home 點的相容邏輯。
- **效果**：解決了因 `seq 0` 為普通 `WAYPOINT` 而實際 `TAKEOFF` 處於 `seq 1` 導致 PX4 飛控可行性檢查器（Feasibility Checker）報出 `Mission rejected: takeoff not first waypoint item` 的 Bug。現在點擊 Plan 介面的 Upload 可順利通過 PX4 校驗並回報 `SUCCESS`。

#### 2. 解決 PX4 飛控不支援任務序列內 RTL 指令之問題 (RTL Command Mapping to LAND)
- **實作結果**：
  - 在 `gateway.py` 的 `_handle_upload_mission` 航點轉譯迴圈中，檢測若 `cmd_str == "RTL"` 且飛控為 PX4，則動態轉譯為 `MAV_CMD_NAV_LAND` (21)。
  - 目標座標優先採用載具最新的遙測 GPS 位置（Home 點或目前位置），若無遙測則 fallback 至任務首個航點，並設定高度為 `0.0`，以模擬安全降落。
- **效果**：解決了 PX4 飛控由於在 Mission 序列中收到 `MAV_CMD_NAV_RETURN_TO_LAUNCH` 而回報 `MAV_MISSION_UNSUPPORTED` (Code: 3) 的拒絕錯誤，現在包含 RTL 航點的任務均能正常完成上傳與同步。

### Feedback
* **問題 1：任務上傳超時 (Timeout waiting for MISSION_ACK)**
  - **根源**：原代碼在傳送完最後一個航點 (`seq == count - 1`) 後立即 `break` 退出接收 `MISSION_REQUEST` 的循環，轉而單獨等待 `MISSION_ACK`。如果最後一個航點封包發生丟包、或者飛控寫入 Flash 較慢而重複請求該航點，Gateway 會因為不再監聽 `MISSION_REQUEST` 而忽略請求，最終導致飛控不發送 `MISSION_ACK` 並引發 Gateway 超時。
  - **修復方案**：改寫上傳等待循環，將 `MISSION_REQUEST`、`MISSION_REQUEST_INT` 與 `MISSION_ACK` 整合至同一個接收循環中。只有在收到 `MISSION_ACK` 時才退出循環；若飛控重複請求某個航點，Gateway 會持續響應，徹底消除由於 Packet Loss 或 Flash 延時引起的上傳超時 Bug。

* **問題 2：無法切入 Auto 模式**
  - **根源**：ArduPilot 的自訂模式編碼與 PX4 完全不同。舊 Gateway 代碼中 `_handle_change_mode` 僅支援 PX4 的 Main/Sub 模式位元移操作（例如 `MISSION = (sub_mode 4 << 16) | (main_mode 4 << 8)`），將此模式值發送給 ArduPilot 會被飛控直接拒絕 (Reject)，導致無法切入 Auto 模式。
  - **修復方案**：在 `_handle_change_mode` 中，根據車輛飛控類型 `msg.autopilot` 進行分支處理。如果是 PX4，繼續採用 Main/Sub 模式位移；如果是 ArduPilot，則直接發送 ArduPilot Copter 標準的模式整數值（`AUTO = 3`、`LOITER = 5`、`RTL = 6`、`LAND = 9`）。

* **問題 3：主畫面無法識別飛控品牌**
  - **修復方案**：
    - **後端**：在 `gateway.py` 接收到 `HEARTBEAT` 訊息時，即時將 `msg.autopilot`（12 為 PX4，3 為 ArduPilot）轉換為對應的品牌字串，並在每 20Hz 的 telemetry 推播封包中新增 `"autopilot": "PX4" | "ArduPilot"` 欄位。同時對 Mock Telemetry 模擬飛機默認添加 `"autopilot": "PX4"`。
    - **前端**：更新 `VehicleTelemetry` 介面類型。在頂部狀態列的 Mode 狀態字樣旁新增精美的 `autopilot-badge` 顯示飛控品牌；在右上角選單中使用者切換下拉選單項目，顯示如 `#1 (PX4)` 或 `#2 (ArduPilot)`；在地圖點擊無人機顯示的 Leaflet Popup 中，於表格內加入一列 `Autopilot` 的實時顯示。

### Report

#### 1. MAVLink 任務上傳超時 (MISSION_ACK Timeout) 的可靠性優化
- **實作結果**：
  - 重構了 [gateway.py](../gateway/gateway.py) 的 `_handle_upload_mission` 核心任務上傳循環。
  - 將本來的分立接收改為統一的事件循環：在 `recv_match` 中同時監聽 `['MISSION_REQUEST', 'MISSION_REQUEST_INT', 'MISSION_ACK']`，並增加 `retries = 5` 與響應超時重發機制。
  - 當飛控仍在發送 `MISSION_REQUEST` 或重複請求最後一個航點時，Gateway 保持持續回覆；只有在真正收到 `MISSION_ACK` 時，狀態機才安全退出。這能 100% 抵禦網絡丟包和飛控寫入 Flash 延遲帶來的 Timeout 錯誤。

#### 2. 多飛控品牌模式切換與心跳解析相容性 (Mode Switch Compatibility)
- **實作結果**：
  - 在 [gateway.py](../gateway/gateway.py) 中，升級 `_mavlink_reader_loop` 中 `HEARTBEAT` 的 `custom_mode` 解析。若為 PX4，使用位移拆解 `main_mode` 與 `sub_mode`；若為 ArduPilot，則通過 `apm_modes` 哈希對照表直接將 Copter 模式代碼轉換成對應的通用狀態字串（如將 `3` 對齊為 `MISSION`，將 `5` 對齊為 `HOLD`），以保證 UI 邏輯的一致性。
  - 在 `_handle_change_mode` 中引入飛控類型動態判定分支。對於 ArduPilot 飛機，發送 `MAV_MODE_FLAG_CUSTOM_MODE_ENABLED` 配合 `custom_mode` 的對應整數值（3 代表 AUTO/MISSION，5 代表 LOITER/HOLD，6 代表 RTL），解決了切入 Auto/Mission 模式被飛控拒絕的 Bug。

#### 3. 前後端飛控品牌識別與 Badge 介面整合 (Autopilot Exposing & Display)
- **實作結果**：
  - **後端數據鏈**：在 [gateway.py](../gateway/gateway.py) 的實時遙測和 Mock Telemetry 模擬循環中，將載具飛控品牌（`PX4` 或 `ArduPilot`）寫入 telemetry `status` 分支，推播給 Web UI。
  - **前端 UI 展現**：
    - 在 [App.tsx](../web-ui/src/App.tsx) 與 [Map.tsx](../web-ui/src/components/Map.tsx) 的數據定義中擴展 `VehicleTelemetry` 與 `MapVehicle` 介面的 `autopilot` 屬性。
    - **頂部狀態列**：在 `qgc-readiness` 區塊（例如 `▶ ARMED — MISSION`）右側動態渲染出一個獨立的 `<span className="autopilot-badge">`（於 [index.css](../web-ui/src/index.css) 中定義其精美的藍色玻璃磨砂 badge 樣式）。
    - **UAV 選單**：切換無人機的下拉選單選項現在會動態拼裝飛控名稱，如 `#1 (PX4)`、`#2 (PX4)` 等。
    - **地圖 Popup**：雙擊或單擊 Marker 彈出的 dynamic card 列表中，加入了一行 `Autopilot: PX4 / ArduPilot` 項目，方便隨時識別當前鎖定載具。
